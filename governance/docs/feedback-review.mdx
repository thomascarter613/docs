# Feedback review

Feedback Review defines how to handle suggestions, corrections, and quality signals from readers, collaborators, and AI agents.

Feedback should improve the docs without allowing unverified changes into canonical pages.

---

## Purpose

Feedback Review exists to:

- capture reader confusion
- identify stale pages
- identify inaccurate claims
- improve search quality
- improve examples
- catch missing links
- review AI agent feedback
- turn feedback into actionable updates

---

## Feedback sources

Feedback may come from:

- page ratings
- comments
- GitHub issues
- edit suggestions
- AI agent feedback
- search behavior
- direct founder notes
- collaborator review

---

## Review process

1. Capture the feedback.
2. Classify it.
3. Check whether it is correct.
4. Identify affected pages.
5. Decide whether to update, reject, or defer.
6. Make the change if approved.
7. Record meaningful changes in the changelog.

---

## Feedback categories

| Category | Meaning |
| --- | --- |
| Correction | Something is wrong |
| Clarification | Something is unclear |
| Missing content | A needed page or section is absent |
| Navigation issue | The page is hard to find |
| Search issue | Search returns poor results |
| Stale content | A page no longer reflects current doctrine |
| AI retrieval issue | An AI system misunderstood or missed context |

---

## Human verification

Human verification is required before acting on feedback that changes:

- accepted ADRs
- governance rules
- pricing
- public claims
- client-facing language
- platform direction
- decision status

---

## Definition of done

Feedback Review is working when feedback becomes a structured improvement loop instead of scattered suggestions.
