# ADR Index

This is the canonical index of all Architecture Decision Records (ADRs) in Monad-Workspace.

Architecture Decision Records preserve the important technical, architectural, operational, and governance decisions that shape Monad-Workspace over time. Each ADR explains what was decided, why the decision was made, what alternatives were considered, and what consequences follow from the decision.

The ADR index is the primary entry point for understanding the architectural lineage of the project.

---

## Purpose

Monad-Workspace is intended to be a local-first, AI-ready, governance-grade, polyglot repo runtime and operating system for modern software work.

Because the project combines runtime architecture, developer experience, native tool coordination, AI context, policy, provenance, graph modeling, and documentation governance, decisions must not remain scattered across chats, commits, issues, or temporary notes.

The ADR system exists to make those decisions:

- explicit
- traceable
- reviewable
- supersedable
- reusable
- understandable to future contributors
- usable by AI-assisted development workflows

An ADR is not merely a note about what was chosen. It is a durable record of why the project moved in a specific direction.

---

## ADR status model

Each ADR should use one of the following statuses.

| Status | Meaning |
| --- | --- |
| Proposed | The decision is being considered but is not yet binding. |
| Accepted | The decision is active and should guide implementation. |
| Superseded | A newer ADR has replaced this decision. |
| Deprecated | The decision is still historically relevant but should no longer guide new work. |
| Rejected | The option was considered and intentionally not adopted. |
| Draft | The ADR exists but is incomplete or not yet ready for review. |

Accepted ADRs are binding until they are superseded, deprecated, or explicitly revised by a later ADR.

## ADR numbering rules

ADR numbers are permanent.

Once assigned, an ADR number should not be reused, even if the ADR is rejected, superseded, or deprecated.

Use this format:

```txt
ADR-0001
ADR-0002
ADR-0003
````

File names should use lowercase kebab-case:

```txt
0001-use-rust-for-monad-runtime.mdx
0002-coordinate-native-tools.mdx
0003-use-multi-crate-rust-workspace.mdx
```

The ADR number identifies the decision. The title explains the decision.

---

## ADR lifecycle

ADRs move through the following lifecycle:

1. **Draft** — the ADR is being written.
2. **Proposed** — the ADR is ready for review.
3. **Accepted** — the ADR is active and binding.
4. **Superseded** — the ADR has been replaced by a later ADR.
5. **Deprecated** — the ADR is no longer recommended but remains historically useful.
6. **Rejected** — the proposed decision was considered and declined.

An ADR should not be deleted merely because the decision changed. The decision history is part of the project history.

---

## When to create an ADR

Create an ADR when a decision affects:

* runtime architecture
* workspace structure
* manifest or configuration design
* CLI behavior
* graph model or output format
* toolchain coordination
* state, cache, lockfile, or provenance behavior
* AI context or memory architecture
* policy, governance, or verification model
* major dependency choices
* developer experience principles
* compatibility or migration strategy

Do not create an ADR for every small implementation detail. Use ADRs for decisions that future contributors or AI agents would need to understand before changing the system.

---

## What every ADR should include

Each ADR should include:

* title
* status
* date
* context
* decision
* alternatives considered
* consequences
* implementation notes
* supersession information
* related documents

A good ADR explains both the chosen path and the rejected paths.

## Review cadence

Review the ADR index monthly.

During review:

* confirm that accepted ADRs still reflect current architecture
* identify decisions that need new ADRs
* mark superseded ADRs clearly
* update links to implementation work
* confirm that the ADR table is complete
* check that new architectural decisions were not left only in chat, issues, or commits

The ADR system should stay current enough to guide real implementation.

---

## Maintenance rules

1. Every ADR must have a permanent number.
2. Every ADR must have a clear status.
3. Accepted ADRs are binding until superseded.
4. Superseded ADRs must link to the replacing ADR.
5. New ADRs should be added to this index immediately.
6. File names should remain stable.
7. ADRs should preserve context, not just conclusions.
8. AI-generated ADR drafts require human review before acceptance.
9. ADRs should be concise but complete.
10. Never delete architectural history merely because the project changed direction.

## Current ADR coverage

The current accepted ADR set establishes the foundation for Monad-Workspace:

* Rust-first runtime
* native tool coordination
* multi-crate workspace organization
* latest stable Rust policy
* early Tokio adoption
* canonical `monad.toml` manifest
* `monad.lock` and `.monad/` state separation
* multi-format graph output

Together, these ADRs define the first architectural baseline for Monad-Workspace.

Future ADRs should build on this baseline rather than re-litigating it without clear new evidence.

---

## Definition of done

The ADR system is working when:

* architectural choices are easy to trace
* future contributors can understand why the system exists as it does
* AI-assisted development can ground recommendations in accepted decisions
* implementation work can reference durable decision records
* superseded decisions remain historically visible
* major technical changes do not happen without recorded rationale

The ADR index is the canonical map of Monad-Workspace architecture decisions.
